## 1. Executive summary

1. **`T.col == None` silently renders `col = $1` bound to NULL — matches zero
   rows instead of NULL rows — and nothing documents or guards it.** Classic
   ORM footgun; produces silently wrong results on a common idiom.
   (`predicate.py:86-90`, `predicate.py:173-177`, `proxy.py:79-83`). **BUG.**
2. **Set-operation operands leak their own `ORDER BY` / `LIMIT` / `OFFSET` /
   `FOR UPDATE` into the middle of the compound statement**, producing invalid
   or semantically-wrong SQL and directly contradicting the documented
   "trailing clauses bind to the whole chain" contract.
   (`executor.py:362-368`, `builders.py:533-563`). **BUG.**
3. **`transaction.__aexit__` discards the user's original exception when the
   `ROLLBACK` itself fails** (no `raise ... from`), and the nested-savepoint
   path issues `ROLLBACK TO SAVEPOINT` without the paired `RELEASE`.
   (`__init__.py:481-501`). **BUG / exception hygiene.**
4. **A large set of SQL constructs is validated only by asserting on FakeDB's
   captured SQL string, never executed against real PostgreSQL** — LATERAL,
   correlated `EXISTS`/`IN`, set-op `ORDER BY`/`LIMIT` scoping, `INTERSECT`/
   `EXCEPT`, several `ON CONFLICT` variants, `UPDATE`/`DELETE … RETURNING`, and
   row locking. This is precisely how findings #1 and #2 ship green. **DESIGN.**
5. **A self-referential foreign key declared before its primary-key field
   raises a misleading "foreign key target has no primary key" error**, because
   `TableMeta.__new__` caches the half-built instance before `pk` is assigned.
   (`meta.py:82-84`, `meta.py:207-212`). **BUG (edge).**

The codebase is otherwise in genuinely good shape — three prior review rounds
closed ~40 findings, and **every prior "CLOSED" item I spot-checked remains
correctly fixed (no regressions).** The new findings are subtle gaps the
earlier passes didn't reach, not backsliding.

## 3. Findings by category

### Correctness

#### BUG — `== None` / `!= None` render as `= $N` / `!= $N` bound to NULL
`predicate.py:86-90`, `predicate.py:173-177`, `proxy.py:79-83`

`ColumnProxy.__eq__` / `Predicate.__eq__` build `Predicate(self, "=", other)`
eagerly. `_render_operand` treats `None` like any plain value:

```python
# predicate.py:173-177
if hasattr(value, "render_sql"):
    result: str = value.render_sql(params)
    return result
params.append(value)
return f"${len(params)}"
```

So `T.col == None` emits `col = $1` with `params = [None]`. Bound through
psycopg that becomes `col = NULL`, which evaluates to `NULL` (never `TRUE`) —
the query matches **zero rows**, and `T.col != None` excludes **every** row.
The library ships `is_null()` / `is_not_null()` as the intended path, but
nothing rewrites, warns, or documents the trap. The insidious case isn't the
literal `== None` (a linter's E711 catches that in user code) but
`T.col == some_var` where `some_var` is `None` at runtime — silent wrong
results, no error, no log. This is the single highest-value finding.

*Fix direction:* in `__eq__`/`__ne__`, when `other is None`, emit `IS NULL` /
`IS NOT NULL` (what SQLAlchemy and Django do — least surprising); or raise at
construction; or, at absolute minimum, add a loud `## Gotchas` entry. The
silent-zero-rows behavior is the worst of the three options.

#### BUG — set-op operands leak ORDER BY / LIMIT / OFFSET / FOR UPDATE into the compound
`executor.py:362-368`, `builders.py:533-563`

The set-op builder methods do nothing but append the operand:

```python
# builders.py:540-542
def UNION(self, other: SelectBuilder) -> SelectBuilder:  # noqa: N802
    self._set_ops.append(("UNION", other))
    return self
```

The executor renders the operand through the *full* select pipeline and
splices it in unparenthesized:

```python
# executor.py:366-368
for op_kw, other in b._set_ops:
    other_sql = self._render_select(other, params)
    sql += f" {op_kw} {other_sql}"
```

`_render_select` unconditionally emits the operand's own `_order` (line 370),
`_limit` (385), `_offset` (391), and `_lock` (399). So
`A.UNION(B.ORDER_BY(B.x).LIMIT(5))` produces
`SELECT … UNION SELECT … ORDER BY x ASC LIMIT 5` — which PG binds to the
*whole* compound, not operand B; and if the left builder *also* carries an
`ORDER BY`, you get two trailing `ORDER BY` clauses → syntax error. An operand
`FOR UPDATE` inside a `UNION` is a hard PG error
("FOR UPDATE is not allowed with UNION/INTERSECT/EXCEPT"). The builder's own
docstring (`builders.py:536-538`) documents the contract this code violates,
and operands are never parenthesized.

*Fix direction:* reject operands that have `_order`/`_limit`/`_offset`/`_lock`
set, or render each operand's body only (SELECT-list/FROM/WHERE/GROUP BY/
HAVING) and forbid those tail clauses on a set-op operand.

#### BUG (edge) — self-referential FK declared before the PK field raises a misleading error
`meta.py:82-84`, `meta.py:207-212`

`__new__` inserts the half-built instance into `_cache` *before* `_introspect`
runs:

```python
# meta.py:82-83
instance = super().__new__(cls)
_cache[target_cls] = instance
```

During introspection, a self-referential FK (`target = self.cls`) resolves
`TableMeta(self.cls)` to that same half-built instance, whose `self.pk` is only
assigned when the PK *field* is reached (line 227-228). If the FK field is
declared above the PK field — e.g. `Node(parent_id: Annotated[int|None,
ForeignKey(Node)]; id: Annotated[int, DBKey])` — then at line 208
`target_meta.pk is None` is true and it raises
`"foreign key target Node has no primary key"`, which is false and misleading.
Field order is legal and unenforced; tree/hierarchy models are the natural
trigger.

*Fix direction:* validate FK targets in a second pass after all fields are
introspected (so `self.pk` is set), or special-case `target is self.cls` to
defer the PK check. A regression test with FK-before-PK ordering should
accompany the fix.

#### DESIGN — bulk INSERT assigns PKs assuming RETURNING row-order == VALUES-order
`executor.py:856-863`

```python
for o, row in zip(b._bulk_objs, rows, strict=True):
    setattr(o, meta.pk.attr_name, row[0])
```

PostgreSQL does not *document* that `INSERT … VALUES (r1),(r2),… RETURNING`
returns rows in VALUES order. Current PG emits insertion order in practice and
this is widely relied upon, but a future planner change or a `BEFORE INSERT`
trigger could reorder, silently mis-assigning PKs to objects. `strict=True`
catches a count mismatch, not a reordering. The integration test that "verifies
RETURNING order" only confirms today's PG happens to preserve it.

*Fix direction:* document the reliance as an accepted assumption (pragmatic),
or correlate rows explicitly. Document, don't over-engineer.

#### SMELL — miss-detection heuristic can misfire on a nullable PK
`executor.py:418-425`

`_object_or_none_if_miss` treats `PK column is NULL` as an outer-join miss. That
holds for a base-table `NOT NULL` PK, but for a CTE/derived "PK" or a `FULL
JOIN` where a matched row legitimately has a NULL in that column, it can't tell
a miss from a real row. Acceptable as a documented heuristic for the common
case; flagging because the docstring states the assumption as if universal.

The docstring promises `async for` usability but `-> Any` gives callers no

### Resources

#### DESIGN — `stream()` server-side cursor relies on async-generator finalization
`psycopg_db.py:116-129`

```python
async def stream(self, sql, params=None):
    async with self._conn.cursor() as cur:
        async for row in cur.stream(self._adapt_sql(sql), params or []):
            yield row
```

This is an async generator holding an open portal inside `async with`. If a
consumer `break`s early and never `aclose()`s the generator, the cursor cleanup
runs only at `aclose()` or loop `shutdown_asyncgens()` — plain GC of an async
generator cannot run async cleanup, so the portal can linger. The builder's
`stream()` adds no `aclosing` either. Mitigated in practice by the
`async with cygnet.transaction(db)` wrapper bounding the portal.

*Fix direction:* document that early-break consumers should wrap in
`contextlib.aclosing(...)`, or have the builder own the lifecycle.

### Exception hygiene

#### BUG — outermost `transaction.__aexit__` masks the original exception on ROLLBACK failure
`__init__.py:494-501`

```python
try:
    if exc_type:
        await self._db.execute("ROLLBACK")
    else:
        await self._db.execute("COMMIT")
finally:
    self._db._in_transaction = False
    self._db._transaction_task = None
```

The `finally` correctly protects flag cleanup, but it does nothing for the
exception. When the body raised and `ROLLBACK` *also* raises (connection
dropped, server already aborted the txn), that new exception propagates out of
`__aexit__` and **replaces** the user's original `exc_val` — with no
`raise rollback_err from exc_val` chaining, the original traceback and cause are
lost. The comment at lines 487-493 only claims to solve the *flag* problem,
which can read as covering the exception path.

*Fix direction:* wrap the `ROLLBACK` in its own `try/except`; on failure chain
(`raise … from exc_val`) or swallow/log so the original `exc_val` re-raises
naturally.

#### SMELL — nested `ROLLBACK TO SAVEPOINT` is never paired with `RELEASE`
`__init__.py:481-484`

```python
if exc_type:
    await self._db.execute(f"ROLLBACK TO SAVEPOINT {self._savepoint}")
else:
    await self._db.execute(f"RELEASE SAVEPOINT {self._savepoint}")
```

In PostgreSQL `ROLLBACK TO SAVEPOINT` rolls back *but leaves the savepoint
defined*. The conventional pattern releases it afterward. As written, the
error path leaves the savepoint on the stack for the rest of the outer
transaction. Usually benign (the outer COMMIT/ROLLBACK discards it), but if the
outer code catches the inner exception and continues, savepoints accumulate.
This branch also has the same exception-masking exposure as the outer path,
without the flag-cleanup safety net.

*Fix direction:* issue `RELEASE SAVEPOINT` after `ROLLBACK TO SAVEPOINT`, and
isolate the savepoint commands from `exc_val` the same way as the outer path.

### Mutability

#### SMELL — `RecursiveCTE.anchor` / `.step` are unchecked public mutables
`cte.py:228-229`

Forward-declaration is deliberate and documented, and the render-time guard
("anchor/step is None → raise") exists. But nothing prevents assigning a
non-`SelectBuilder` (a raw string, a plain `CTE`) — a bad assignment surfaces as
an `AttributeError` deep in the executor rather than at the assignment site,
unlike `exists()` / `JOIN_LATERAL` which `isinstance`-check at construction.
Low priority; consider a validating property setter or accept the documented
"fail loud later" stance.

### API design

#### SMELL — `ON CONFLICT` action methods silently clobber a prior action
`builders.py:814-838`

`DO_UPDATE` / `DO_NOTHING` / `DO_UPDATE_FROM_EXCLUDED` use
`dataclasses.replace(spec, action=…)` with no guard that an action was already
set. So `.ON_CONFLICT(c).DO_UPDATE(x=1).DO_NOTHING()` silently downgrades
update→nothing (and leaves a stale `set_kwargs` in the spec), and the reverse
flips nothing→update. Contrast the *deliberate* second-call rejection in
`_set_lock` (`builders.py:466-471`). Action re-specification is silent and
undocumented.

*Fix direction:* reject re-setting a non-`None` action, or document it as
intentional clobber like the single-slot INTO/SET/FROM clauses.

#### SMELL — `VALUES()` doesn't guard against an established `_select_source`
`builders.py:732-744`

`BULK_VALUES` guards against `_select_source`, and `SELECT` guards against
`_obj`/`_kwargs`/`_bulk_objs`, but `VALUES` only checks `_bulk_objs`. So
`.SELECT(src).VALUES(obj)` sets `_obj` alongside `_select_source`; the executor
checks `_select_source` first (`executor.py:583`) and silently drops the
`VALUES` object. The four-way mutual-exclusion the docstring claims has this one
hole.

*Fix direction:* add the symmetric `if self._select_source is not None: raise`
to `VALUES`.

#### DESIGN — `FOR_UPDATE(of=…)` accepts tables not present in the query
`builders.py:481-485`, `executor.py:399`

`_set_lock` type-checks each `of` entry but never verifies it's the FROM table
or a joined table. `FOR UPDATE OF not_joined_table` is a runtime PG error
("relation … in FOR UPDATE clause not found in FROM clause") that surfaces only
at execution — defeating the client-side-rejection philosophy applied to
`nowait`/`skip_locked`.

*Fix direction:* validate each `of` table against `{FROM} ∪ {joins}` at
`_set_lock` time, or document the gap.

### Performance

#### SMELL — `meta.fields.index(meta.pk)` recomputed per row of every outer join
`executor.py:420`

`_object_or_none_if_miss` runs a linear `.index()` scan over `meta.fields` once
per joined-table chunk per result row to find a position that is constant for
the table — O(rows × fields) avoidable work on the hot row-mapping path.

*Fix direction:* compute the PK index once (cached property on `TableMeta`, or
hoisted before the per-row loop) and pass it in.

### Tests

#### DESIGN — high-risk SQL constructs are validated only by FakeDB string assertions
`tests/test_lateral.py`, `tests/test_subquery.py`, `tests/test_set_ops.py`,
`tests/test_on_conflict.py` (`TestOnConflictRender`), `tests/test_windows.py`
(frame strings), `tests/test_locking.py`, `tests/test_builders.py`
(UPDATE/DELETE RETURNING)

The integration suite is genuinely strong for core DML and the type matrix.
But the constructs most prone to *silent malformation that a substring match
accepts* are unit-only:

- **LATERAL / `LEFT JOIN LATERAL`** — `$N` inner-then-outer threading and the
  `) alias ON true` syntax asserted only against FakeDB. **No integration test
  executes a LATERAL.** Highest-risk gap.
- **Correlated `EXISTS` / `NOT EXISTS` / `IN (SELECT …)` / scalar subquery** —
  no integration coverage of any correlated subquery.
- **Set-op `ORDER BY`/`LIMIT` scoping, `INTERSECT`, `EXCEPT`, `UNION ALL`,
  chained set-ops** — integration covers only plain `UNION`. (This is exactly
  the area of executive finding #2 — the scoping bug would pass the existing
  substring-order unit assertion.)
- **`ON CONFLICT ON CONSTRAINT`, multi-column target, `DO_UPDATE_FROM_EXCLUDED`,
  `DO_UPDATE`-kwargs `$N` numbering** — unit-only.
- **`UPDATE … RETURNING` / `DELETE … RETURNING`** row mapping — unit-only.
- **Row locking** (`FOR UPDATE`/`FOR SHARE`/`OF`/`NOWAIT`/`SKIP LOCKED`) — a
  documented selling point, never executed.

*Fix direction:* add one round-trip integration test per construct above. They
are cheap (the harness exists) and would catch the class of bug this very
review found by hand.

#### SMELL — comparison bench `== 100` assertion depends on class-definition order
`bench/comparison/test_comparison.py:216` vs `:252`/`:296`

`TestSelectAll::test_cygnet` asserts `len(rows) == 100`, but `TestInsertOne` and
`TestBulkInsert` insert into the *same* module-scoped table (pytest-benchmark
runs each op many rounds → hundreds of rows) with no re-seed between classes. It
works only because pytest runs classes in definition order; a reorder or
`pytest-randomly` breaks it.

*Fix direction:* make the assertion `>= 100`, or give the insert benchmarks a
throwaway table / function-scoped re-seed.

#### SMELL — comparison bench overstates SA/Cygnet INSERT equivalence
`bench/comparison/test_comparison.py:268-279` vs `:255-263`

Cygnet's `InsertOne` reuses a module-scoped autocommit connection; the SA path
constructs a fresh `AsyncSession` and `await s.commit()` *inside* the timed op.
The comment calls these "equivalent," but SA pays session-construction +
explicit-commit cost Cygnet doesn't. The pool clamp (`pool_size=1,
max_overflow=0`) is correct and fair; only the equivalence comment overstates.

*Fix direction:* soften the comment, or hoist session creation out of the timed
region.

#### NIT — dead `pytest.warns` statement
`tests/test_bench_compare.py:142`

A bare `pytest.warns` attribute-access on its own line is a no-op with a
misleading comment about silencing ruff F401. Remove it.

### CI / automation

#### SMELL — `bench._summary` is untested and crashes on malformed JSON
`bench/_summary.py:23`, `tests/` (absent)

```python
for b in data["benchmarks"]:
    ...
    ops = 1 / b["stats"]["median"]
```

`data["benchmarks"]` is unguarded (`KeyError` on a JSON missing the key) and
`1 / median` is a latent `ZeroDivisionError`. Its sibling `bench._compare` is
well covered by `tests/test_bench_compare.py`; `_summary` has zero tests despite
running in CI to populate `$GITHUB_STEP_SUMMARY` (`ci.yml:259`). The realistic
risk is low (a real pytest-benchmark JSON always has the key and non-zero
medians, and an empty `benchmarks` list just yields a header-only table), but
the asymmetry with the tested sibling is the smell.

*Fix direction:* add a `render` test mirroring `test_bench_compare.py`,
including the missing/empty-`benchmarks` case; guard the key access.

The CI pipeline itself is **well-constructed**: ruff format-check + ruff lint +
mypy strict + pytest on 3.12/3.13, integration across PG 14–18, a build job, a
weekly + per-push `pip-audit` (correctly non-editable-skipping), and an
advisory `continue-on-error` benchmark job with PR-vs-main delta. This matches
the project's stated CI conventions. No CI gap to file.

### Docs vs reality

- **`== None` trap undocumented** — folds into the correctness BUG above. The
  `## Gotchas` section lists several NULL-related caveats but not this one.
- **Set-op docstring documents a contract the code breaks** —
  `builders.py:536-538` states trailing `ORDER BY`/`LIMIT` bind to the compound;
  the executor's operand rendering makes that false whenever an operand carries
  its own tail clauses. Folds into executive finding #2.
- Otherwise the README and `CLAUDE.md` are unusually accurate to the code — the
  prior review rounds clearly kept them in sync.

### Security

No issues. The "operators and function names are trusted strings; operands are
always parameterized" model is applied consistently — every value operand flows
through `params.append()` → `$N`, and the helper modules
(`functions`/`jsonb`/`arrays`/`fts`) never route user input into an operator or
function-name position. The one raw-SQL escape hatch (`lit()`) is documented as
trusted, and the `$N`→`%s` rewrite in `psycopg_db` is documented as
string-literal-blind with the caveat reproduced on `lit()`'s docstring. The
`OFFSET`/`LIMIT` integer interpolation rejects negatives and is never
user-controlled.

## 4. Open questions

1. **Is `T.col == None` → `IS NULL` rewriting wanted, or is the explicit
   `is_null()` path a deliberate design choice?** If deliberate, the silent
   zero-rows behavior still needs a loud `## Gotchas` entry at minimum; the
   current silence is the problem regardless of which way the API decision goes.
2. **Are set operations intended to support per-operand `ORDER BY`/`LIMIT`
   (i.e. `(SELECT … LIMIT 5) UNION (SELECT … LIMIT 5)` with parenthesized
   operands), or only compound-level tail clauses?** The fix differs: parenthesize
   operands vs. reject operand tail clauses.
3. **Is the bulk-INSERT reliance on RETURNING row-order an accepted assumption?**
   If yes, it should be a one-line comment at `executor.py:856`; if no, the
   correlation needs a design decision.
4. **What's the intended field-ordering contract for self-referential FK
   models?** Either enforce PK-first, or make introspection order-independent
   (two-pass FK validation).
