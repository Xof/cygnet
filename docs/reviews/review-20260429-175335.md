## Executive summary

1. **Stale doc reference for the reference adapter.** `README.md:618` directs readers to `tests/integration/conftest.py:PsycopgDB` for the streaming reference — but PsycopgDB now lives at `cygnet/psycopg_db.py`. The conftest reduces to a re-export. Casual readers chasing the link land at a one-line `# noqa: F401` instead of the real adapter.
2. **Regex-based `$N` → `%s` translation has a footgun for `lit()`.** `cygnet/psycopg_db.py:51` translates *every* `$\d+` in the SQL string, including ones embedded inside `cygnet.lit("'$1 some text'")`. The escape hatch is documented as trusted, but the translator's blindness to string literals isn't called out — a user passing pre-numbered placeholders inside `lit()` will silently have them rewritten.
3. **`InsertBuilder.ON_CONFLICT_*` is a six-method state machine validated piecemeal.** Five `_on_conflict_*` fields, six methods, cross-validation embedded in each method body. It works and is well-tested, but the implicit state machine is the kind of surface where a future addition (e.g., `ON CONFLICT … DO UPDATE … WHERE …`) will introduce a subtle bug because the rules live in six places.
4. **Executor module-level imports could be lifted.** `cygnet/executor.py:117/181/419/481` use function-local `from .cte import …` / `from .proxy import ColumnProxy` imports with the comment "avoid the circular dep". The cycle isn't there: `cte.py` does its `proxy` import lazily already, so executor → cte → proxy is acyclic at module load time. Same dead-weight pattern that builders.py recently removed (`# verified-acyclic now, the inline imports were dead weight`).
5. **`save()` doesn't refresh the in-memory object on the upsert path.** `cygnet/executor.py:864-932` — for the existing-PK INSERT … ON CONFLICT DO UPDATE branch, no RETURNING is emitted, so trigger-modified or generated-column values stay stale on the in-memory object. The fresh-INSERT branch *does* refresh via run_insert. Inconsistent and the docstring doesn't flag it.

## Findings by category

### Correctness

**SMELL** `cygnet/psycopg_db.py:50-60`
The translation regex `r"\$\d+"` rewrites every `$N` substring without distinguishing SQL syntax from string-literal contents. A user invoking `cygnet.lit("WHERE name = '$1 prefix'")` gets `WHERE name = '%s prefix'`, and the unmatched `%s` then either errors at psycopg or — worse — silently consumes a real param. The comment on lines 45-50 documents position-insensitivity but not literal-blindness.
*Direction of fix*: Either narrow the regex to only match outside string-literal contexts (proper but invasive), or document the limitation explicitly in the `lit()` docstring and `psycopg_db.py` header. The library's stance (lit is trusted) makes the second acceptable.

**SMELL** `cygnet/executor.py:864-932` (run_save)
The upsert path emits `INSERT … ON CONFLICT (pk) DO UPDATE SET non_pk = EXCLUDED.non_pk` with no `RETURNING`. After `await cygnet.save(db, obj)`, the in-memory object holds whatever the caller passed in — but if the table has triggers, generated columns, or `DEFAULT now()` on update, the DB row diverges from `obj`. The fresh-INSERT branch (line 889-896) goes through `run_insert` which *does* refresh via RETURNING, so the same `save()` call surfaces fresh data on first save and stale data on update.
*Direction of fix*: Either always emit `RETURNING *` and `setattr` every column, or document the divergence on `save()`'s docstring. The current "save() always sends the full object" line in README is necessary-but-not-sufficient — it tells the user about the request, not the response.

**NIT** `cygnet/executor.py:42-69` (`_check_predicates`)
Three branches that interact non-obviously: empty list + SELECT → `[]`, empty list + UPDATE/DELETE → raises, non-empty all-True → `[]`, mixed → raises, real-only → returns `real`. The control flow could be clearer with the empty-list-with-real branch returning `predicates` directly (instead of falling through to the post-filter return).
*Direction of fix*: Cosmetic. Optional re-order — extract the "compute filtered list" first, then validate.

### Typing

**SMELL** `cygnet/builders.py` (53 `Any` occurrences)
A handful are necessary (`db: Any` for duck typing); many are speculative. `_on_conflict_target: tuple[Any, ...] | None` could be `tuple[ColumnProxy[Any], ...] | None` since the executor immediately checks `isinstance(c, ColumnProxy)` at render time (`cygnet/executor.py:425-432`). `_on_conflict_excluded` is the same. `_obj: Any` on InsertBuilder/UpdateBuilder is technically heterogeneous (any dataclass instance), but a `Protocol` for "dataclass with a TableProxy" would catch wrong-table objects at the call site rather than the executor.
*Direction of fix*: Tighten the on_conflict tuple types. The `_obj: Any` story is harder; defer.

**NIT** `cygnet/builders.py:142`
`self._columns = columns` assigned without a type annotation; relies on `*columns: SQLRenderable` typing through. Inconsistent with sibling fields that carry explicit annotations.
*Direction of fix*: `self._columns: tuple[SQLRenderable, ...] = columns`.

**NIT** `cygnet/cte.py:21-37` (`_PseudoField`)
`primary_key: Any = None` and `foreign_key: Any = None` defeat the type checker for CTE column attribute access. The fields are documented as "the executor only reads attr_name / column_name / primary_key / foreign_key". A `Protocol` defining that minimal shape — `FieldLike` — would let `_PseudoField` satisfy it without the `Any` defaults.
*Direction of fix*: Define a `FieldLike` Protocol in `expression.py` (or a new `protocols.py`) and have FieldMeta + _PseudoField both satisfy it.

**SMELL** `cygnet/cte.py:122-134` (CTE's TableMeta-shaped surface)
CTE returns `self` from `_meta` and exposes `table_name`, `fields`, `pk`, `cls` properties to mimic TableMeta. RecursiveCTE duplicates the same surface. The `# type: ignore[arg-type]` on `setattr(self, col, ColumnProxy(self, field))` (line 73) is a tell: CTE is structurally a TableProxy-like but isn't typed as one. A `TableSourceProtocol` (Protocol with the four properties) would let TableProxy, CTE, and RecursiveCTE all advertise a common interface.
*Direction of fix*: Extract the protocol; use it as the union member type in `TableSource = …` and as the return type of `_meta`.

### Async / sync boundary

### Exception hygiene

**NIT** `cygnet/psycopg_db.py:71-74`
```python
try:
    return await cur.fetchall()
except psycopg.ProgrammingError:
    return []
```
ProgrammingError is broad. The intent is "the statement returned no result set" (DML without RETURNING, DDL, etc.) and psycopg uses ProgrammingError for that case — but a future psycopg version could narrow or change this. `cur.description is None` is a deterministic test for "no result set" without depending on exception semantics.
*Direction of fix*: Replace with `if cur.description is None: return []` plus a `return await cur.fetchall()` else-branch. Same outcome, no exception-driven control flow for the "I asked something that doesn't return rows" case.

### Mutability

No findings. Every cross-cutting type that should be immutable is `@dataclass(frozen=True)` (PrefixOp, SuffixOp, _LockClause, Literal, _Column, _PrimaryKey, _ForeignKey, _PseudoField, _Exists, FunctionCall, WindowSpec, WindowExpression). Mutable defaults are absent.

### API design

**SMELL** `cygnet/builders.py:614-690` (InsertBuilder ON CONFLICT cluster)
Six methods (`ON_CONFLICT`, `ON_CONFLICT_CONSTRAINT`, `ON_CONFLICT_DO_NOTHING`, `DO_NOTHING`, `DO_UPDATE`, `DO_UPDATE_FROM_EXCLUDED`) sharing five `_on_conflict_*` state fields, with cross-validation duplicated across method bodies (e.g., `if self._on_conflict_constraint is not None: raise …` in ON_CONFLICT, the symmetric check in ON_CONFLICT_CONSTRAINT). Five state fields means 32 possible bit-states; five methods that toggle subsets means the reachability isn't obvious from the code. Currently safe because the test suite covers the matrix, but adding a sixth dimension (e.g. `WHERE` clause on the UPDATE branch, which PG supports) would force the bit-flag dance out of one more method.
*Direction of fix*: A single `_OnConflictSpec` frozen dataclass — `target: Target | None`, `action: Action | None`, `set_kwargs: dict | None`, `excluded_cols: tuple | None` — with a class method per builder method that returns a new spec. Validation collapses into the dataclass's `__post_init__`.

**SMELL** `cygnet/expression.py:107-140` (`op` overloads)
Three behaviours under one name (1-arg factory, 2-arg prefix, 3-arg infix), distinguished by arity at runtime via `len(args)`. The `@overload` decorations type-check correctly, but the runtime body's `if len(args) == 3: ... elif len(args) == 2: ...` requires a reader to keep all three semantics in mind to follow any one branch. The 1-arg factory-factory is rarely used in tests (grep finds only one direct use); whether it earns its keep is worth asking.
*Direction of fix*: Optional. If the 1-arg form is rarely used, deprecate it and let users write the trivial `lambda l, r: op(l, "ILIKE", r)` themselves.

**NIT** `cygnet/builders.py:469-472` (`EXCEPT_`)
Trailing underscore because `except` is reserved. Documented in the README. The corresponding non-set-op verbs (UNION, INTERSECT) don't have the underscore. Inconsistent at the API surface; minor.
*Direction of fix*: None — the language constraint is real.

### Concurrency

**SMELL** `cygnet/__init__.py:230-308` (`transaction`)
The class docstring (lines 233-260) and the README's "Concurrency caveat" both call out the `_in_transaction` flag's non-task-locality. Strong documentation, but the runtime offers no guard: a user who reuses one PsycopgDB across two asyncio.tasks will silently corrupt nesting. A `__aenter__` check that fingerprints the current task (`asyncio.current_task()`) and raises if it diverges from the one that flipped `_in_transaction = True` would catch the misuse loudly.
*Direction of fix*: Optional but worth considering. The fingerprint check costs a single `asyncio.current_task()` call per nesting boundary, and the failure mode it prevents is hard to debug.

### Imports / structure

**NIT** `cygnet/executor.py:117/181/419/481`
Function-local imports of `RecursiveCTE`, `Lateral`, `ColumnProxy` with the comment "avoid the circular dep" (the comment doesn't appear at every site, but the pattern's rationale is consistent). Verified that the cycle isn't real:
- `executor.py` imports `from .proxy import TableProxy` at module level (line 33) — proxy.py imports from `.predicate`, no cycle.
- `cte.py` imports `from .proxy import ColumnProxy` *inside its constructor* (line 64-66), so cte.py's module-load order doesn't pull proxy in.
- `executor.py` could safely add `from .cte import RecursiveCTE, Lateral` at module level.

The pattern matches the dead-weight `builders.py` removed in an earlier commit (per its own header comment: "verified-acyclic now, the inline imports were dead weight").
*Direction of fix*: Lift the three local cte/proxy imports to module level. The `from .builders import InsertBuilder` on line 890 *is* a real cycle (builders → executor at module level), so leave that one in place.

**NIT** `cygnet/executor.py:481` and `cygnet/executor.py:419`
Two separate `from .proxy import ColumnProxy` calls inside two methods. Even before the lift suggested above, these duplicate imports could be a single one inside whichever ancestor the call sites share — but lifting to module level is cleaner.

### Docs vs reality

**SMELL** `README.md:618`
The streaming section says: `psycopg's cursor.stream() is the reference implementation (see tests/integration/conftest.py:PsycopgDB)`. The actual PsycopgDB lives at `cygnet/psycopg_db.py`. `tests/integration/conftest.py` does re-export it (lines 19-20: `from cygnet.psycopg_db import PsycopgDB  # noqa: F401`), so the path technically resolves — but the canonical home is the package module. Readers chasing the citation hit a one-line re-export.
*Direction of fix*: One-line edit — change to `cygnet/psycopg_db.py:PsycopgDB`.

**SMELL** `cygnet/__init__.py:122-129` (`lit()` docstring)
The docstring says "The SQL is emitted verbatim — no escaping, no parameter substitution. … the string is trusted." True at the Cygnet layer, but in combination with `cygnet/psycopg_db.py`'s `$N`→`%s` regex, a `lit()` containing a `$1` substring (e.g., `cygnet.lit("'$1'")`) gets translated. The docstring doesn't mention that downstream-adapter assumption.
*Direction of fix*: Add a sentence noting that the reference psycopg adapter rewrites `$\d+` patterns and that lit() consumers should avoid those substrings. Or accept the limitation and don't mention.

### Packaging

**Strengths**: `py.typed` marker present (`cygnet/py.typed`); strict mypy; tight version floors (`psycopg[binary]>=3.1`, `pytest>=8.0`, etc.); no upper bounds (deferred per REVIEW.md).

**NIT** `pyproject.toml:23` — `psycopg[binary]>=3.1` is a runtime dependency, but Cygnet *itself* doesn't import psycopg except in `cygnet/psycopg_db.py`. A user shipping their own adapter would still pull in psycopg. The codebase comments call out psycopg_db as "the only place Cygnet itself imports psycopg" — the dep listing should reflect that. Move `psycopg[binary]>=3.1` to an optional `psycopg` extra and gate `psycopg_db.py`'s import behind it; the core install becomes psycopg-free.
*Direction of fix*: `dependencies = []`, `[project.optional-dependencies] psycopg = ["psycopg[binary]>=3.1"]`, and either keep `psycopg_db.py` importable only when the extra is installed (raise a helpful ImportError otherwise) or move it under `cygnet.adapters.psycopg`. Mind the path-breakage cost.

## Open questions

1. **`save()` upsert refresh.** Was the lack of RETURNING * on the existing-PK path a deliberate trade-off (cost of an extra column round-trip vs staleness) or an oversight? The docstring is silent.
2. **The `op()` 1-arg factory-factory.** Used in tests? Used by anything in the wild? If it's purely speculative, simplifying the overload set would tighten the public API.
3. **`psycopg[binary]` dependency placement.** Is the psycopg dep intended to be required (making Cygnet effectively a psycopg-coupled library) or optional (making PsycopgDB a sample adapter)? README and code comments lean toward optional, but pyproject.toml says required.
4. **`tests/integration/conftest.py` re-export.** Is the re-export there for back-compat with old test imports, or because PsycopgDB really did live there once? Either way, the README citation should point at the canonical home.
