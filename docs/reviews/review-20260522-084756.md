## Executive summary

1. **`PsycopgDB.column_defaults` lookup ignores schema.** `WHERE table_name = $1` against `information_schema.columns` matches *every* schema's tables, not just the one in the connection's search_path. A deployment with two `events` tables (`public.events`, `archive.events`) would pull the union of both tables' DEFAULT columns, and Cygnet would silently omit columns that don't have DEFAULTs in the table actually being INSERTed into.
2. **`run_save` still doesn't refresh — and now also suppresses DEFAULTs.** The May 17 commit added DEFAULT-aware omission and RETURNING to `run_insert` / `run_create`, but `run_save`'s upsert branch was deliberately excluded. Result: `save()` on `created_at=None` against `created_at DEFAULT now()` writes NULL on insert and clobbers to NULL on conflict. The prior review's OQ1 (no RETURNING refresh) is unresolved; a sibling DEFAULT-suppression bug now compounds it.
3. **`_defaults_cache` has no schema-change invalidation hook.** The cache is permanent for the adapter's lifetime. A migration that adds, removes, or alters a DEFAULT mid-process will be invisible to Cygnet's INSERT codegen until the adapter (i.e., the connection) is reconstructed. Long-running services that hot-reload schema will silently produce wrong INSERTs.
4. **`stubs._format_type` loses generic parameters.** `getattr(t, "__name__", None)` returns `"list"` for `list[str]` (parameterised generics still carry `__name__`), so generated stubs emit `ColumnProxy[list]` instead of `ColumnProxy[list[str]]`. Surfaces the moment any project declares a list/dict column and runs `python -m cygnet.stubs`.
5. **Dependency audit silently swallows CVEs.** `pip-audit --strict || true` in `.github/workflows/ci.yml:168` exits 0 regardless of strict-mode failures — the `--strict` flag is decorative. A real CVE in psycopg or a dev tool surfaces only in the job log, not as a failing build.

## Findings by category

### Correctness

**BUG** `cygnet/psycopg_db.py:144-151` (`column_defaults`)
The SQL is `SELECT column_name FROM information_schema.columns WHERE table_name = %s AND column_default IS NOT NULL`. No `table_schema` filter. `information_schema.columns` exposes columns from *every* schema visible to the role; the query returns the union across schemas. The accompanying comment at lines 131-136 says: *"Looks up the unqualified table name in the connection's current schema search_path (no explicit `table_schema` filter)."* That is false. `WHERE table_name = $1` does not consult `search_path`; it returns matches from every schema. A deployment with `public.events DEFAULT now() on created_at` and `archive.events` (no DEFAULTs) inserting into `archive.events` would have Cygnet omit `created_at` because the cross-schema lookup says it's a DEFAULT-bearing column. PG then either uses no default (column has none in this schema) writing NULL — wrong row data — or, if `archive.events` lacks `created_at` entirely, raises a column-not-found error.
*Direction of fix*: Add `AND table_schema = ANY(current_schemas(false))` (PostgreSQL's view of the resolved search_path), or use `pg_catalog.pg_attribute` joined to `pg_class` filtered by `relnamespace` to get a single-schema answer that honors the search_path properly.

**BUG** `cygnet/executor.py:55-114` (`_defaults_cache`, `_get_defaulted_columns`)
The cache lives on the Executor *class* keyed by adapter instance. There is no expiration, no invalidation API, and no version tag. The class-level comment (lines 62-67) says "The answer is stable for the lifetime of the schema." That's true — but the cache lives for the lifetime of the *adapter*, which is typically the lifetime of the connection, which can easily outlive a schema migration. A long-running service that pulls a connection from a pool, INSERTs into `events` (caching `{created_at}`), then a migration drops the DEFAULT, then the same connection INSERTs again, will keep omitting `created_at`. PG then writes NULL where the application expects the historical default to fire.
*Direction of fix*: Provide a `flush_column_defaults()` adapter method or a class-level cache-clear utility. Document the staleness window in `PsycopgDB.column_defaults`'s docstring. A timestamp/TTL is heavier-handed; a manual flush is enough for migration-aware deployments.

**BUG** `cygnet/executor.py:1186-1263` (`run_save`)
The upsert branch (the existing-PK path) emits `INSERT … ON CONFLICT (pk) DO UPDATE SET non_pk = EXCLUDED.non_pk` with every column emitted at full value. No `column_defaults` consultation, no RETURNING. The code comment at lines 1199-1207 acknowledges the divergence ("does NOT participate in the DEFAULT-aware column-omission machinery"), but the README's "save() always sends the full object" (line 345) doesn't surface this to users. Consequence: `save()` on `Account(id=existing, name="x", created_at=None)` against `created_at DEFAULT now()` writes NULL on first save (already covered by Apr 29 #5) AND, on conflict, the EXCLUDED.created_at update clobbers the existing row's `created_at` to NULL.
*Direction of fix*: Either route the upsert through `_extract_insert_fields(defaulted=…)` like `run_insert` does, and emit RETURNING for the same shape; or document the user-visible semantics in the README and on `save()`'s docstring explicitly ("save() emits every column at its current Python value; columns with schema DEFAULTs are written as NULL if the in-memory value is None"). The former is the correct behaviour; the latter is at least honest.

**BUG** `cygnet/stubs.py:126-136` (`_format_type`)
`getattr(t, "__name__", None)` returns `"list"` for `list[str]`, `"dict"` for `dict[str, int]`, etc. Python 3.12 parameterised generics carry `__name__` on the origin class. The function's intent (per docstring lines 128-132) is that union and parameterised types fall through to `str()`; the code only achieves this for `UnionType` (which has no `__name__`). Running `python -m cygnet.stubs myapp.models` with a model containing `Annotated[list[str], …]` produces `ColumnProxy[list]` — mypy then sees the attribute as `ColumnProxy[list[Unknown]]` and any operation expecting `list[str]` elements fails to type-check.
*Direction of fix*: `if isinstance(t, type) and not hasattr(t, "__class_getitem__"): return t.__name__`; or simpler, `if type(t) is type: return t.__name__` since parameterised generics have `type(...) is types.GenericAlias` (or `_GenericAlias` for typing.X variants). Test coverage in `tests/test_stubs.py` doesn't currently exercise a parameterised-generic field — that's why CI is green.

**SMELL** `cygnet/executor.py:1267-1286` (`_row_to_obj`)
`zip(meta.fields, row)` silently truncates on length mismatch. If the row is longer than `meta.fields`, the trailing columns are dropped without warning. The docstring (lines 1276-1279) documents the truncation in both directions. The renderer guarantees `len(row) == len(meta.fields)` for the implicit-columns case, so this only bites with `cygnet.lit()` columns that smuggle in extra positions, or schema drift on `SELECT *`-style paths (which Cygnet doesn't currently emit). Not a regression; documented; flagged because the failure mode (silently-wrong objects) is severe relative to the warning surface.
*Direction of fix*: Replace `zip` with `zip(meta.fields, row, strict=True)` and let the resulting ValueError surface a meaningful error.

**SMELL** `cygnet/builders.py:_HavingMixin / SelectBuilder.HAVING` (~line 380, exact location varies)
The `HAVING` docstring claims `cygnet.all` is rejected, but no `isinstance(predicate, _All): raise` guard exists in the method body. A caller can pass `cygnet.all` and the predicate flows through to render time. Probably harmless (renders as the `_All` sentinel's render_sql, typically nothing or TRUE), but the docs-vs-reality gap is the finding.
*Direction of fix*: Add the explicit isinstance check, or relax the docstring.

### Exception hygiene

**SMELL** `cygnet/executor.py:104-112` (`_get_defaulted_columns`)
```python
try:
    self._defaults_cache[self._db] = per_table
except TypeError:
    return result
```
The bare `TypeError` catch is narrow in *practice* (WeakKeyDictionary raises TypeError only on non-weak-referenceable objects), but the breadth of the catch could mask future bugs introduced into the cache-population path. Catching `TypeError` from a single specific operation is fine; catching it over a multi-statement block where new code might also raise TypeError is the pattern to avoid.
*Direction of fix*: The current shape is a single statement, so this is more "watch out as you grow" than a current bug. Worth a comment noting which specific TypeError is expected.

### API design

**SMELL** `cygnet/psycopg_db.py:111-151` + `cygnet/executor.py:_get_defaulted_columns` (the `column_defaults` optional method)
The duck-typed db adapter contract used to be three methods: `execute`, `execute_one`, `stream`. The May 17 commit added a fourth, `column_defaults`, as an *optional* method probed via `hasattr`. This is functional but starts a pattern that doesn't scale well: optional protocol methods that change codegen behavior when present create a many-axes-of-adapter-compatibility matrix (does this adapter omit DEFAULTs? does it know about RETURNING shape? etc.). Today there's one optional method; if a similar trick is added for, say, schema introspection or RETURNING shape, the protocol becomes a documentation problem.
*Direction of fix*: Formalize the protocol with a `typing.Protocol` and a `runtime_checkable` decorator. Document the optional methods explicitly. Consider exposing capabilities via a single `adapter_capabilities() -> set[str]` method instead of `hasattr` probes — that way the adapter has one place to declare what it can do.

### Concurrency

**SMELL** `cygnet/executor.py:96-114` (cache-miss race)
Two concurrent INSERTs against the same (adapter, table) — which is documented as unsupported, since the adapter isn't task-safe — would both miss the cache, both call `column_defaults`, and the second writer's `per_table` would clobber the first's. Even within the documented "one adapter per task" pattern, two sibling `asyncio.gather` tasks each holding their own adapter can race on the *class-level* `_defaults_cache`: each task creates its own `per_table` dict for its own adapter, no contention. But if the user reuses one adapter across tasks (which they shouldn't), the second task's `per_table = {}` initialization wipes the first task's already-cached entries.
*Direction of fix*: This isn't a real bug given the documented contract, but the failure mode is silent. A `setdefault` would be enough: `per_table = self._defaults_cache.setdefault(self._db, {})`.

### Performance

**NIT** `cygnet/executor.py:75-114` (`_get_defaulted_columns`)
First INSERT to a never-seen table now pays one synchronous round-trip for the `information_schema.columns` lookup. Cached thereafter. For an app that boots, INSERTs into N tables once each, then never again, the cost is N round-trips. Acceptable; documenting because it's a behavior change vs. the pre-May-17 contract.

### CI / automation

**BUG** `.github/workflows/ci.yml:168` (`audit` job)
```yaml
- name: Run pip-audit
  run: pip-audit --strict || true
```
The `|| true` swallows pip-audit's exit code unconditionally, including the strict-mode failures the `--strict` flag opts into. A real CVE in psycopg or any dev dep produces a non-zero exit which the shell discards, the job passes, no notification. The job comment at lines 144-147 calls this "non-blocking by design" — fine — but the strict flag is then dead weight. Either drop the `|| true` (strict-mode failures become blocking, which is the point of `--strict`), or drop `--strict` and accept that the audit is informational.
*Direction of fix*: Drop one or the other. Recommend dropping `|| true` for `--strict` — the strict mode is exactly the case where you want to block.

**NIT** `.github/workflows/ci.yml` overall
No `schedule:` trigger. The audit job runs only on `push` to main and on `pull_request`. CVE-in-transitive-deps that fires *between* PRs (which is common — supply-chain vulnerabilities don't sync to your commit cadence) surfaces only when someone happens to push. A weekly `cron` trigger on the audit job is the standard mitigation.
*Direction of fix*: Add `schedule: - cron: "0 4 * * 1"` (Monday 04:00 UTC) and gate the audit job on that plus push/PR.

**Strength**: PG 14/15/16/17/18 integration matrix is exemplary; few libraries this size cover that range. Codecov upload, hatch build, bench advisory all in one workflow.

### Docs vs reality

**SMELL** `cygnet/psycopg_db.py:131-136` (`column_defaults` docstring)
*"Looks up the unqualified table name in the connection's current schema search_path (no explicit `table_schema` filter)."* The implementation does not consult `search_path`; it returns matches from every schema. See Correctness BUG #1. This docstring needs to either describe what the code actually does ("matches the table_name across every schema visible to the connection — undefined behaviour with same-named tables in multiple schemas") or the code needs to be fixed to match the docstring.

**SMELL** `cygnet/psycopg_db.py:138-143`
The comment mentions `pg_get_expr(adbin, adrelid)` returning the rendered DEFAULT expression — but the code uses `information_schema.columns.column_default IS NOT NULL`, not `pg_get_expr`. The comment is informative-by-analogy ("here's how PG represents it under the hood") but a reader expecting to see `pg_get_expr` in the code below will be confused.

**SMELL** `README.md:345` ("save() always sends the full object")
True at the request level but misleads on response/semantics. save() now diverges from create()/INSERT in two ways: no DEFAULT-aware omission, no RETURNING refresh. A user reading this line will assume save() is the obvious uppercase mass-update; they'd be wrong about both DEFAULT behavior and in-memory state after the call.
*Direction of fix*: Add a sentence to save()'s README block: "Columns with schema DEFAULTs are written as NULL on save() — DEFAULTs only fire when save() routes through the initial INSERT path (DBKey with None). Use INSERT or create() if you need DEFAULTs to fire for an existing PK."

**SMELL** `CLAUDE.md` JOIN family list
Lists `RIGHT_JOIN` and `FULL_JOIN` as part of the JOIN family. Neither exists in `cygnet/builders.py` or `cygnet/executor.py`. Already flagged in `docs/reviews/comment-run-20260522.md:Finding #6`; restating for completeness.
*Direction of fix*: Trim CLAUDE.md, or implement them. Implementing both is one line each in the JOIN cluster.

### Packaging

**SMELL** `uv.lock` is checked in; `pyproject.toml` and `justfile` use pip, not uv
The presence of `uv.lock` (~committed) without an accompanying `uv`-driven workflow is a tooling-ambiguity smell. Either:
- `uv` is the canonical resolver and the lockfile is authoritative — in which case `just bootstrap` should `uv sync`, not `pip install`, and CI should use uv too;
- pip is canonical and the lockfile is incidental — in which case it should be `.gitignore`d.

Today the repo has both: pip everywhere, lockfile sitting in version control out of band. A future contributor sees the lockfile and assumes uv; tests with pip succeed; environments diverge.
*Direction of fix*: Pick one. If keeping uv, switch bootstrap to `uv pip install -e ".[dev]"` and update CI's install step. If dropping it, remove the lockfile.

### Security

**SMELL** `cygnet/__init__.py:lit()` (line 122-129)
`lit()` is documented as trusted; users can pass `cygnet.lit(user_input)` and Cygnet won't escape. That's the explicit contract. Combined with the `$N`→`%s` rewrite (Apr 29 #2 still open), there's a layered footgun: a user passing `cygnet.lit("'$1 prefix'")` ends up with the literal value's `$1` rewritten to `%s`, which then either errors at psycopg or silently shifts param indices. Both `lit()`'s docstring and `psycopg_db.py`'s header should call out the interaction.
*Direction of fix*: One-sentence addition to `lit()`'s docstring: "When used with the psycopg adapter (or any adapter that translates `$N` placeholders), avoid embedding `$<digit>` substrings — they will be rewritten."

## Open questions

1. **Is the DEFAULT-aware path intended for `run_save` eventually, or is the divergence permanent?** The code comment at `cygnet/executor.py:1199-1207` reads as a deliberate choice ("consistent with save() semantics") but the README doesn't surface it and the divergence between sibling methods is the kind of thing that bites users in production. Worth deciding whether to formalize the divergence (document it loudly) or close the gap.
2. **Should the duck-typed adapter contract be promoted to a formal `Protocol`?** The May 17 introduction of `column_defaults` as an optional method changes the shape of the protocol. With one optional method, `hasattr` is fine. With two, the pattern starts to feel like a capability bitmap. Decide now while the surface is small.
3. **Is `uv.lock` authoritative, advisory, or accidental?** See packaging finding above. The presence of a lockfile without a uv workflow is the canonical "is this still in scope" tooling signal.
4. **Should the audit job's `--strict` block on CVEs, or should the audit be informational?** Today the `--strict || true` combination is contradictory. Either is defensible; the current combination isn't.
